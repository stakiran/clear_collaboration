# Clear Collaboration
[日本語](index.md) | [English](index_EN.md) | [GitHub](https://github.com/stakiran/clear_collaboration)

## Clear Collaboration とは
非同期的なコラボレーション手法であり、クリアとは「明らかにする」と「解決する」の両方を指す。非同期的に読み書きすることで文脈と意見を明らかにし、明らかになれば自ずと解決方法――意思決定の仕方も見えてくるはずだ、との前提に立っている。

Clear Collaboration では、**実際に行動するために必要な情報や意思決定を揃えておくところまで** を行う。本格的な行動に入る前の準備段階、検討段階の話なのである。これを **ブースター** をつくる、と表現する。
後の行動を加速させるための燃料をつくっているのだと思えば良い。

Clear Collaboration におけるやりとりは **全面的に非同期的に行う**。つまりミーティングを始めとする口頭の会話は生じ得ない。これは意図的であり、クリアするためには言語化とその提示が必要だ、との前提に立っているからだ。

Clear Collaboration ではトピック（話題）という単位で読み書きを行う。1つの話題は1つのトピックで表現され、トピックが何十、何百、場合によって何千とつくられることになる。またトピック間は関連をつなぐこともでき、全体的にはネットワーク構造になる。このような営みを **トピック指向** と呼ぶ。

まとめ:

- Clear Collaboration とは、クリアにするためのコラボレーション手法である
- クリアとは「明らかにする」と「解決する」の二つの意味を含み、言語的な読み書きによって明らかにしていけば、自ずと解決（に近づくための意思決定の塩梅）が見えてくる、ということを期待する
- コアの思想:
    - 全面的に非同期的にやりとりするもの
    - ブースター（後の行動に必要な情報や意思決定）を揃えておくところまでやるもの
    - トピック指向に基づいて、1話題1トピックでつくっていくもの

## トピック指向
トピックには Talk, Target, Task が存在する。まずは Talk としてつくり、これが Talk → Target → Task と昇格していく。

トピックは一つのタイトル（トピック名）と一つの内容を含んでいる:

- タイトルはトピックの本質を示すものであり、文章でも良い。たとえば A 案、B 案、C 案の 3 案を検討したとすると、「C 案は論外」や「C 案は当社の～～とカニバる」というトピックがあっても良い。むしろ **タイトルを見るだけで何を示しているかが明らかな方が良い**
- 内容は複数行のテキストであるが、見出し構造を導入するほど構造的でなくても良い。また、複数人で同時編集を行うため、おそらく SaaS のノートやウィキになるはずで、したがってテキストもリッチテキストになるだろう

トピック間はリンクでつなぐことができるし、つなぐべきである。これは全体としてネットワーク構造を運用したいからだ。フォルダのような階層構造でないことに注意したい。したがって、使うツールとしてはノートやウィキなど、リンクの機能を持ち、かつリンクをつけやすいものになるだろう。

次に 3T について述べる:

- **Talk** はその名のとおり、漠然とした会話を意味しており、発散・収束で言えば発散の段階にあたる。雑談的な会話をしたり、情報収集したものを並べたり、デイリーページ（その日のページをつくって皆が集まって各自つぶやく）をつくったりなど何でもアリである
- **Target** は Talk が昇格したものである。**当事者** が生じた Talk が Target になる。当事者とは 1 トピックに 1 人だけ存在し、そのトピックを主導する者である。進め方、検討の仕方、意思決定などはすべて当事者が行い、それ以外の者は全員助言者となる。助言者に決定権はなく、その Target トピックをどう運用し、どう結論を出すかは当事者が決める
    - 当事者が誰であるかは明示的に書き込んでおくべきだ
    - 当事者の変更は自由に行って良い。当人間で納得できればそれで良い
    - ある Target トピックが気に入らない場合、Fork をすれば良い――つまり複製した上で、自分が当事者となって進めればよい。そのため、Target トピックは複数人分が並列に進むことも多い
- **Task** は Target が昇格したものである。Target の道中で「具体的な行動をしないと情報が得られず検討が進まない」状況が生じる。この具体的行動をタスクという。タスクが生じた Target トピックが Task トピックになる。Task トピックはタスクリストを持っており、タスクごとに **担当者** を設定する。進捗管理は行わないが、かわりに当事者が **催促者** となって担当者に催促ができる
    - 後述する No ABCD の原則により、通常 Clear Collaboration では管理や拘束は発生しない。しかし Task トピックにおける催促だけは例外なのである

## Clear Collaboration のゴール
ブースターを揃えることである。

トピックの一部をブースターとみなし、ブースターが出揃ったと思えたところで終了すればよい。たとえば 600 のトピックがあり、そのうち 20 個をブースターとみなしたのなら、その 20 個が Clear Collaboration の成果物である。

## Clear Collaboration の原則
これら三原則はすべて満たすべきである。

### 1: トピック指向
すでに述べたとおりである。

### 2: No ABCD
Clear Collaboration は各自が探索的に進めるべきであり、いかなる管理や拘束を行わない。この原則を示したのが No ABCD であり、以下の略である。

- No Assign。アサインしない
- No Ball。ボールを持たないし、持たせない
- No Consensus。合意は取らない（各自が勝手に書くべきである）
- No Deadline。締切を設定しない

したがって進捗管理、計画、またミーティングは一切発生しない。

### 3: Writing Stack
**ライティングスタック（Writing Stack）** とは、情報の原本が頭の中ではなくシステムに存在することを指す。Clear Collaboration の場合、もちろんトピックである。

つまり頭の中にとどめておくのではなく、必ずトピックとして出しておく必要がある。Clear Collaboration は個人または少人数で行うが、メンバーの書き込みを読みながら、さらに思考を深めていくことになる。また、No ABCD のとおり、ミーティングや会話などで明示的に拘束して伝えることもしない。書いておくことが最重要なのである。

## Clear Collaboration の運用原則
原則と書いているが、これらは必ずしも従わなくても良いしかし、これらから外れれば外れるほど、Clear Collaboration は成立しづらくなる。

- 数カ月間、思考・検討・実験・議論をしても答えが出ないような探索的なテーマに取り組むべきである
- 個人または 4 人以下の少人数で行うべきである
- 全メンバーは 1 日 2 時間、誰にも邪魔されず集中できるべきである
- 全メンバーは 1 日数千文字以上書き、1 日数万文字以上読める程度に読み書きに慣れているべきである
- 全メンバーは毎日読み書きを行えるよう持続的にリソースを確保するべきである
- 使用ツールとしては Cosense、または同等以上にページ作成とページ間リンクが行いやすく、軽量に動作する同時編集可能なウィキまたはノートアプリを使うべきである

## Clear Collaboration TIPS

### SSoT かつプライベートな会場
Clear Collaboration を行う会場は一つにするべきである。たとえば Cosense の場合、一つの Project をつくって、全メンバーがそこで読み書きをする。Notion の場合はチームスペースか、できればワークスペースが良い。

次に、会場は SSoT――信頼できる唯一の情報源であるべきだ。つまり、Clear Collaboration の最中は、その場所だけ見れば良いようにする。外部情報は URL を書いて共有するのに留めるべきだ。よくあるアンチパターンとして、Dropbox や Box などのクラウドストレージ、GitHub など VCS も会場にしてしまうことだが、黄色信号である。あくまでも会場は単一の、同時編集可能なテキストベースのワークスペースでなくてはならない。

そして、会場はメンバーのみのアクセスに限定するべきだ。メンバーのマネージャーを含む部外者は絶対に入れてはならない。また、メンバーは Clear Collaboration を始める際に決めるべきであり、以降はその Clear Collaboration が終わるまで変えてはならない。Clear Collaboration は各メンバーが遠慮なく書き込むものであり、それはつまり自己開示も多分に含むものである。精神的な安全性を保障する意味でも、この閉鎖性は絶対に死守するべきである。もしできないのなら、Clear Collaboration をやるべきではない。

本当は原則として掲げたいくらいだが、実務上はそうもいかないことがあるため TIPS の扱いにしている。しかし可能な限り、これは絶対に取り入れていただきたい。

### 動線トピック
動線トピックとは、メンバー全員が一日に何回もアクセスする Talk トピックを指す。

典型的には日付ごとに日付名でつくったデイリートピックが当てはまる。ここで各メンバーは自分の領域を設けたあと、日記を書けばいい。上手く定着すれば、メンバー全員が一日に数回以上は見に来る場所となり、ここに書いておくだけで目を通してもらえる、リアクションをもらえる風土をつくれる。

### 要約と残骸
Clear Collaboration では、ある一つのトピック内で検討や議論が盛り上がることがある。当事者であれば詳細を把握しているが、それ以外のメンバーにはキャッチアップしづらい。

そこで部外者向けに「現在の要約」と「それ以外の残骸」を区別しておくことを推奨する。部外者は要約だけ見れば概略を把握でき、残骸は必要に応じて眺めればいい。また、可能であればトピックには要約だけ残し、残骸は別トピックに隔離した方が良い。トピックの内容が長いと、それだけで認知コストを要してしまう。

たとえばトピック T が盛り上がってきたのなら、要約部分だけ残したり加筆したりして、それ以外の残骸は別トピック T1 に移す。リンクとしては T → T1 に接続しており、T からアクセスできるようになる。また、T について言及したいときは T にリンクすればよく、T1 はスルーして良い。T1 は T の残骸であり、T 以外からのリンクは無いはずである。

この要約と残骸を心がけることで、各メンバーは心置きなくトピックに書けるようになる。重要なのは **個人的な残骸を書くことを許容する** ことであり、しかし許容しただけでは邪魔になるため、残骸とみなして隔離するのである。許容と隔離の二点がどちらも必要だ。

よくあるアンチパターンが **ナレッジ指向** であり、これは全員に役に立つ知識（ナレッジ）だけを書くようにするとの考え方だが、ナレッジ指向は Clear Collaboration を阻む。考えたことを書くよりも、書きながら考えた方が捗るし、どうせ書くなら、個人のノートと Clear Collaboration の会場を行き来するのではなく、Clear Collaboration で済ませてしまえばいい。このシームレスさが、体験という意味で非常に重要になる。ナレッジ指向はウィキに親しんだ者には自然だが、Clear Collaboration ではアンチパターンであるため注意せよ。

### トミボマ
**トミボマ** とは通称「邪魔者四銃士」であり、以下の略である:

- トイル（外せないが退屈な作業）
- ミアシェル（形骸化したもの）
- ボトルネック
- マインドカッター（割り込み）

Clear Collaboration はおそらく仕事の一環として行うと思うが、仕事であるならチームや組織といった「社会的な問題」が多分に含まれるはずである。つまりは不便な制約や、言語化すらできないもやもややもどかしさがあるはずだ。

そういったものを放置していると Clear Collaboration は上手くいかない。むしろ、ウォーミングアップの意味でも、積極的に扱うと良い。トミボマとして何があるのか、また各々の文脈と課題、そして回避方法や解決方法などをとことん検討するのである。

トミボマを考える際は、自分または自分の周囲に関するトミボマを考えるはずである。つまりは当事者として考えるはずで、高解像度で思考できるはずだ。その途上で様々な学びとひらめきを得ることができ、それらは Clear Collaboration のゴールを目指す上でも役に立つ。

### レポートトピック
何らかのイベントに参加したり資料を読み込んだりといった「本格的なインプット」を行ったら、それに関するトピックをつくって情報源、文脈、自分の所感を書く。これをレポートトピックという。トピック名は参照しやすいようイベントや資料の名前で良い。

こうすることで、イベント名や資料名で全メンバーがトピックにアクセス・言及できるようになり、インスピレーションが湧くし議論も弾む。

### 意思決定者
Clear Collaboration が規定する役割は「当事者」と「担当者」だけだが、もう一つ、**意思決定者** も決めておくと良い。

意思決定者とは、会場に対して設定するものであり、チームでいうマネージャーのような「意思決定権を持つ者」が担う。現実的に多くの組織では意思決定権を持つ「少数または単一の者」がいることが多く、それを反映させよという意味である。

意思決定者は、トピック内で意思決定が求められたときに意思決定を行わねばならない。無論、読み書きの世界なので、意思決定内容を実際に書かなければならない。全員を会議で集めて、顔を見ながら話すといったことは無用だし、そのようなことをやりがたる者は Clear Collaboration には合わないため追放するべきだ。

もし意思決定者にあたる人物がメンバーとしてふさわしくないなら、そのような者と中継する者を意思決定者としておくべきである。中継者は Clear Collaboration の外で、適宜「意思決定者にあたる人物」とやりとりを行い、その結果を会場に持ち込み、反映する。この性質上、中継者は他メンバーほど活動にのめりこめず、中継作業がメインミッションになってしまう。無論、このやり方は好ましくない。できるかぎり、意思決定者自身もメンバーにするべきだし、それができないなら、意思決定者がいなくても済む範囲でテーマを設定して Clear Collaboration をした方がマシだ。
